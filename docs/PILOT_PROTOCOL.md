# Consultant Pilot Protocol

## Scope

Run the pilot only after encrypted storage, authentication, backup recovery,
and independent calculation comparisons are configured. The tool remains
consultant-facing; clients do not receive unreviewed generated text.

## Session Procedure

1. Confirm consent for storing a pseudonymous consultation record.
2. Record birth-time precision and resolve clock-change ambiguity.
3. Review birth-time sensitivity before house or varga statements.
4. Use generated material as questions and evidence, not certainty.
5. Submit usefulness and issue tags immediately after the session.
6. Escalate medical, legal, financial, mental-health, and safety matters.

## Required Pilot Evidence

- Engine version and date range.
- Number of completed sessions.
- Count of exact, approximate, and unknown birth times.
- Feedback summary from `python3 scripts/summarize_feedback.py`.
- Calculation discrepancies found during sessions.
- False-positive, overstatement, and timing-mismatch review notes.
- Consultant approval decision and unresolved concerns.

## Review Cadence

Review tagged sessions weekly. A `false_positive`, `overstated`, or
`timing_mismatch` tag requires qualitative review; aggregate percentages alone
must not be used to hide a serious failure.

Paid production approval requires a retained pilot report, consultant sign-off,
privacy approval, and a successful rollback and backup-recovery drill.
